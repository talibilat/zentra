Serve mode backend readiness

This report describes the backend that is implemented in AgentTrail 0.1.0, the existing core behavior it reuses, the serve-specific behavior it introduces, and the boundaries that remain intentionally local and process-scoped.

Readiness summary

Ingestion

Ready for one JSONL file or standard input. Files are followed after EOF, partial lines are held until newline completion, and replacement or truncation is reported.

Projection

Ready for graph, tree, swimlane, sequence, inspector, warning, usage, and playback views. Layout coordinates remain client-owned.

Streaming

Ready through typed Server-Sent Events with monotonically increasing cursors, snapshot reconciliation, heartbeats, and reconnect recovery.

Security

Loopback by default. Remote binding requires an explicit flag, generated token access, and rejects unsafe unredacted mode.

New serve-mode behavior

Source lifecycle

Separates source connectivity from trace completion. File EOF means caught up; stdin EOF means disconnected; only trace.completed and trace.failed terminate runs.

Run projection

Produces agents, event offsets, usage totals, primary parents, causal links, message links, unresolved endpoints, source findings, runtime warnings, and lifecycle state.

HTTP API

Serves an offline shell, versioned snapshots, run details, lazy payload details, and SSE from a standard-library threaded HTTP server.

Browser application

Provides graph, tree, swimlane, sequence, playback, inspectors, filters, warning history, progressive reveal, and live-follow behavior without runtime third-party assets.

Known approximations

  • Representative span per actor: one actor ID represents one logical invocation, and projected parentage uses the first causal cross-actor span relationship that introduces the actor.
  • Fallback run titles: run identity is the sanitized trace ID; no separate title contract is implemented.
  • Process-local run list: all runs come from the current input stream and current server process.
  • No cross-restart history: there is no database or persistence layer.
  • Warning resolution: runtime warning history is observed while snapshots are requested; there is no separate durable warning journal.
  • Dense rendering: the client initially renders 80 agents and offers progressive reveal rather than automatic clustering.

Verification evidence

Automated coverage launches the real serve command, drives packaged Chromium UI with Playwright, appends to a growing file, verifies SSE and reconnect recovery, exercises all visual views and investigation controls, checks sanitization and compatibility, and measures first useful paint with 100 agents.

The redaction ruleset is intentionally bounded. Users must still review browser output before sharing trace data.
